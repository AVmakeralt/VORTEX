## VORTEX: The Gap Between JIT and C — Full Optimization Analysis

VORTEX already has an impressive architecture: 4-tier compilation, Sea-of-Nodes IR, partial escape analysis, adaptive guards, and ML-guided inlining. But being "as fast as C" means eliminating **every** layer of abstraction cost between the source semantics and bare metal. Here's what's missing, ranked by impact.

---

### 1. 🔴 Type Specialization with Speculative Untagging (HIGHEST IMPACT)

**The Problem:** Every arithmetic operation currently pays the NaN-boxing tax. Even in JIT-compiled code, `IADD` requires:
```
check tag == SMI → extract int64 → add → re-encode as SMI
```
That's **3-5 extra instructions per operation** vs. C's single `add`. For a tight loop doing integer math, this alone accounts for a **3-5x slowdown** vs. C.

**The Fix — Speculative Untagging:**
- At T2/T3, emit **unboxed fast paths**: assume SMI, do raw `add r64, r64`, guard on overflow and type
- Only re-box at deopt/materialization points
- Use **type feedback from T0/T1** to know which locals are always SMI/always double

```c
// Current (what you have): every IADD does this
if (vtx_is_smi(a) && vtx_is_smi(b)) {
    result = vtx_make_smi(vtx_smi_value(a) + vtx_smi_value(b));
}

// Target (what you need): JIT emits this
// mov rax, [rbp+local_i]    ; load raw bits
// add rax, [rbp+local_j]    ; direct add on tagged SMI (if shifted encoding)
// jo overflow_handler        ; single overflow guard
```

**Implementation Strategy:**
- Change SMI encoding to **left-shifted by 1** (like V8) instead of the current NaN-box shift-by-3. This makes `a + b` on tagged SMIs produce a tagged SMI result *without any encoding/decoding* — the add is the tagging. This single change eliminates the tag/untag overhead for integer math entirely.
- For floats, use a **dedicated double array** in FrameState (like LuaJIT's split-stack design): one `uint64_t[]` for tags, one `double[]` for values. When the JIT knows a local is always a double, it skips tag checks and operates on the raw double.
- Add a **TypeSpecialization** IR pass after SCCP/GVN that replaces `Add(Int, Int)` with `AddI32`/`AddI64` nodes that don't encode/decode.

### 2. 🔴 Optional GC / Manual Memory Management Mode

**The Problem:** The GC touches every layer of the runtime — write barriers on every field store, safepoint checks at every backward branch, root stack pushes/pops, and pinning overhead for JIT raw pointers. This is 10-20% overhead even when you don't need it.

**The Fix — `vtx_gc_mode_t`:**

```c
typedef enum {
    VTX_GC_GENERATIONAL,  // current behavior
    VTX_GC_NONE,          // no collection, no barriers, no safepoints
    VTX_GC_MANUAL,        // explicit alloc/free, no barriers
    VTX_GC_ARENA,         // bump-pointer only, free-all-at-once
} vtx_gc_mode_t;
```

**What changes in each mode:**

| Component | Generational | None | Manual | Arena |
|-----------|-------------|------|--------|-------|
| `vtx_gc_alloc` | bump-pointer + check | bump-pointer only | malloc-like | bump-pointer only |
| `vtx_gc_write_barrier` | full check | **no-op** | **no-op** | **no-op** |
| `vtx_gc_safepoint` | check + maybe collect | **no-op** | **no-op** | **no-op** |
| `vtx_gc_root_push/pop` | push to root stack | **no-op** | **no-op** | **no-op** |
| JIT guard emission | guard + deopt stub | guard + deopt stub | guard + deopt stub | guard + deopt stub |

The key insight: make `vtx_gc_write_barrier`, `vtx_gc_safepoint`, and `vtx_gc_root_push/pop` **function pointers** initialized at startup based on the mode. In `VTX_GC_NONE` mode, they point to no-op stubs. The JIT then **inlines the no-op** and dead-code-eliminates the calls entirely.

For `VTX_GC_MANUAL`, add `VT_OP_FREE` opcode and `vtx_gc_manual_free()`. The allocator uses a free-list (like the current old gen) so individual frees are O(1).

For `VTX_GC_ARENA`, provide `vtx_gc_arena_enter()` / `vtx_gc_arena_leave()` which set a bump-pointer and free everything at leave. This is the fastest possible allocation mode — literally one instruction (`lea`) per allocation.

### 3. 🔴 Missing Compiler Optimizations (Critical IR Passes)

The current IR optimization pipeline is: **SCCP → GVN → DCE → Schedule**. This is good but far from what production JITs do. Missing passes, by impact:

#### 3a. Loop-Invariant Code Motion (LICM)
**Impact: HIGH** — Without LICM, every loop recomputes address calculations, field loads, and type checks that don't change across iterations. C compilers hoist these automatically.

**Implementation:** After scheduling, walk loop bodies (identified by `LoopBegin`/`LoopEnd`). For each non-pinned, non-side-effecting, non-memory node whose inputs are all defined outside the loop or are loop-invariant, move it to the preheader. Guard nodes can also be hoisted (you already have `guard/hoist.h` but it's limited to guards — extend it to all data nodes).

#### 3b. Induction Variable Strength Reduction
**Impact: HIGH for array-heavy code** — Currently `ARRAY_LOAD` at offset `base + i * 8` recomputes the multiply each iteration. C compilers replace this with a pointer that increments by 8.

**Implementation:**
- Detect linear induction variables: `i = init + step * trip`
- Replace `base + i * scale` with `ptr = base + init * scale; ptr += step * scale` per iteration
- The SoN IR already has `Add`, `Mul`, `Shl` nodes — detect the pattern and rewrite

#### 3c. Loop Unrolling (beyond the `loop_spec` skeleton)
**Impact: MEDIUM-HIGH** — Eliminates branch overhead, enables instruction-level parallelism, and exposes more CSE opportunities.

**Implementation:**
- Use profile trip-count data (already collected by T0/T1)
- For trip counts with CV < 20% (already detected in `sota/phase.c`), unroll by the known count
- Otherwise, unroll by 2-4x with a pre-loop guard
- **Critical:** After unrolling, re-run GVN + DCE to eliminate redundancies exposed by duplication

#### 3d. Bounds Check Elimination
**Impact: HIGH for tight loops** — Every `ARRAY_LOAD`/`ARRAY_STORE` currently emits a bounds check guard. C has no bounds checks.

**Implementation:**
- When the index is an induction variable and the loop bound ≤ array length, eliminate the bounds check entirely
- Use the SoN IR's `Guard` nodes: if `Guard(i < len)` dominates `Guard(i < len)` with the same `len`, the dominated guard is redundant
- Already partially handled by guard merging, but needs explicit range analysis

#### 3e. Alias Analysis / Memory Disambiguation
**Impact: HIGH** — Without alias analysis, the scheduler must be conservative: every `Store` blocks subsequent `Load`s even if they access different objects. C compilers know `a->x` and `b->y` don't alias.

**Implementation:**
- Add a **type-based alias analysis (TBAA)** pass: attach `type_id` + `field_offset` metadata to `LoadField`/`StoreField` nodes
- Two memory accesses `LoadField(obj1, offset=3, type=Point)` and `StoreField(obj2, offset=7, type=Line)` cannot alias
- Use this in the scheduler to allow `Load` to move past `Store` when they provably don't conflict

#### 3f. Register-Based Bytecode (Alternative: Direct IR from Source)
**Impact: MEDIUM** — Stack-based bytecode has inherent dispatch overhead: every operation requires stack manipulation. Register-based bytecode (like LuaJIT's) encodes operands directly, reducing instruction count by ~30%.

**Implementation option 1:** Design a register-based bytecode format as an alternative compilation target. V8 went this route with Ignition (register-based interpreter).

**Implementation option 2 (better):** Skip bytecode entirely for T2+ and go directly from trace recording to SoN IR. The trace recorder already walks bytecode — instead of recording bytecode PCs, directly emit SoN nodes. This eliminates the bytecode→IR translation overhead entirely.

### 4. 🟡 Object Layout & Allocation Optimizations

#### 4a. Object Header Compression
**Current header:** 24 bytes (type_id:4 + gc_mark:1 + gc_age:1 + gc_pinned:1 + gc_remembered:1 + size:4 + shape_id:4 + field_count:4 = 20 bytes, padded to 24). That's a lot of overhead for small objects.

**Fix:** In `VTX_GC_NONE` mode, the header can shrink to just `type_id:4 + shape_id:4 + field_count:4 = 12 bytes` (padded to 16). The GC fields are only needed when GC is active. Use a **mode-dependent header size** — the JIT knows the mode at compile time and adjusts field offsets accordingly.

#### 4b. Allocation Folding
When PEA eliminates allocations (scalar replacement), the object never materializes. But when it can't fully eliminate (partial escape), the allocation should still be optimized: fold multiple allocations into a single bump of the allocation pointer, then initialize all objects. This reduces allocation overhead from N pointer bumps to 1.

#### 4c. Inline Caching in Baseline JIT (T1)
Currently T1 doesn't use inline caches — it does full vtable walks for every `CALL_VIRTUAL`. Add **inline cache slots** in the JIT frame (like V8's vector ICs): at each call site, store the last-seen target. On the next call, compare the receiver's shape_id, and if it matches, call directly. This turns virtual dispatch from O(vtable depth) to O(1) for monomorphic sites.

### 5. 🟡 Code Generation Quality

#### 5a. Smarter Register Allocation
The current linear scan is good but has a key weakness: it doesn't consider **register hinting**. When a value is produced in RAX (e.g., from a `CALL`) and consumed as the first argument (which also needs to be in RDI), the allocator should preferentially assign RDI to that value early.

**Fix:** Add **affinity-based coalescing** to the register allocator. Before allocation, compute move hints: if a value must move from vreg_A to vreg_B (due to ABI constraints), try to assign them the same physical register to eliminate the move.

#### 5b. Two-Address Form Optimization
x86-64 is a two-address architecture: `add rax, rbx` clobbers `rax`. Currently, if the SoN IR says `c = a + b` and `a` is live after the add, you must emit `mov rcx, rax; add rcx, rbx`. With **copy elision**: if the scheduler can prove `a` is dead after the add, reuse `a`'s register directly, saving the `mov`.

#### 5c. Conditional Move Elimination of Branches
Short if-else chains like `x = cond ? a : b` should use `cmov` instead of branch+mov. Currently you emit `CMP + JCC + MOV + JMP + MOV`. C compilers emit `CMP + CMOV`. Add a **pattern matcher** in instruction selection that detects `If(Region(Proj(cond, true), Proj(cond, false))) → CMOV`.

---

### 6. 🟡 Concurrency & Threading

#### 6a. Thread-Local Allocation Buffers (TLABs)
Currently the bump-pointer allocator is single-threaded by design. For multi-threaded use, each thread needs its own TLAB — a small chunk of the young gen carved off with a single atomic increment. Allocation is then completely lock-free within each TLAB.

#### 6b. Concurrent GC
The current stop-the-world GC pauses all mutator threads. For low-latency workloads, implement **concurrent marking** (like ZGC/Shenandoah): the mutator runs while the marker traces, using a read barrier (Brooks pointer or load barrier) to handle object moves. This is complex but essential for "as fast as C" in server workloads.

### Priority Implementation Order

If you want the maximum performance gain for minimum effort, implement in this order:

1. **Speculative untagging + left-shifted SMI** — eliminates 60-80% of overhead on integer-heavy code. This single change will bring integer loops within 1.5-2x of C.
2. **Optional GC modes** — removes write barriers and safepoints entirely in `VTX_GC_NONE` mode. Another 10-20% on object-heavy code.
3. **LICM + bounds check elimination** — makes tight loops competitive with C's unguarded loops.
4. **Type-based alias analysis** — enables the scheduler to move loads past stores, critical for C-like memory performance.
5. **Induction variable strength reduction** — eliminates redundant multiplies in array accesses.
6. **Affinity-based register coalescing** — reduces register shuffling, especially around call sites.

After these 6 changes, VORTEX should be within **1.1-1.3x of C** on pure numeric code with `VTX_GC_NONE`, and within **1.5-2x** on general object-oriented code with GC enabled. The remaining gap comes from deopt guards (which C doesn't have) and the inherent cost of dynamic dispatch (which is opt-in in C via vtable).

## The Bombs — Research-Level Optimizations

### BOMB 1: Superoptimization via Enumerative Search

**Source:** Bansal & Aiken, "Automatic Generation of Peephole Superoptimizers" (OSDI '06); Sasnauskas et al., "Souper" (CGO '17)

**The Idea:** Instead of pattern-matching known optimization cases, **synthesize** the optimal instruction sequence for hot basic blocks by brute-force enumerating all possible x86-64 instruction sequences up to length N, checking equivalence via SAT/SMT, and keeping the fastest one.

**Why It's a Bomb:** Production compilers miss enormous numbers of peephole optimizations. Souper found optimizations that LLVM missed in **17% of hot basic blocks** in real C++ programs. A JIT can apply this only to the 0.01% of code that actually matters (the hottest 10-50 basic blocks), making the exponential cost tractable.

**Implementation:**
```
1. Profile identifies top-50 hottest basic blocks (by execution count)
2. Extract the block's SoN IR subgraph + type constraints
3. Send to a synthesizer backend (Z3 or custom SAT encoder):
   - Encode the block's semantics as SMT constraints
   - Enumerate candidate sequences from an instruction grammar
   - Check equivalence, measure estimated latency (using uops.info tables)
4. Replace the block's instruction sequence with the synthesized one
5. Cache the optimization pattern for future matching (learning)
```

**Expected gain:** 5-15% on hot integer loops. Some synthesized sequences are 30-40% shorter than what even the best hand-tuned instruction selectors produce.

**Scalability:** Only applied to blocks exceeding a heat threshold (e.g., >10M executions). The synthesizer runs in a background thread with a time budget. For large repos, this targets the 80/20 rule: 80% of execution time in 0.01% of code.

### BOMB 2: Polyhedral Loop Optimization

**Source:** Feautrier, "Dataflow Analysis of Array Programs" (1992); Bondhugula et al., "PLuTo" (CC '08); Tavares et al., "PPCG" (2018)

**The Idea:** Model loop nests as integer polyhedra (systems of linear inequalities over iteration variables and array indices). Then apply **loop tiling, interchange, fusion, skewing, and parallelization** — the same transformations that make BLAS/LAPACK fast.

**Why It's a Bomb:** No production JIT does this. HotSpot C2 has basic loop opts. V8 has nothing. GraalVM has nothing. This is the domain of research compilers and auto-parallelizing compilers. A JIT uniquely knows the **actual trip counts and array sizes**, which lets it choose tile sizes that are optimal for the real L1/L2 cache dimensions.

**Implementation:**
```
1. Identify hot loop nests (already done by trace selector)
2. Check if the loop nest is affine (indices are linear functions of iterators)
   - If yes: model as polyhedron using isl (Integer Set Library) or a lightweight custom implementation
   - If no: fall back to standard loop opts
3. Compute optimal tile sizes using actual cache parameters from CPUID + measured trip counts
4. Generate transformed SoN IR: tiled, interchange-optimized, with prefetches
5. If iterations are independent: generate parallel loop (see Bomb 3)
```

**Example transformation:**
```c
// Before (typical matrix multiply):
for (i = 0; i < N; i++)
  for (j = 0; j < N; j++)
    for (k = 0; k < N; k++)
      C[i][j] += A[i][k] * B[k][j];

// After polyhedral optimization (tiled for L1 cache):
for (ii = 0; ii < N; ii += TILE)
  for (jj = 0; jj < N; jj += TILE)
    for (kk = 0; kk < N; kk += TILE)
      for (i = ii; i < min(ii+TILE, N); i++)
        for (j = jj; j < min(jj+TILE, N); j++)
          for (k = kk; k < min(kk+TILE, N); k++)
            C[i][j] += A[i][k] * B[k][j];
```

**Expected gain:** 2-10x on dense array/matrix loops. Near-zero overhead on non-affine code. For large repos, this only fires on hot numeric kernels.

### BOMB 3: Speculative Thread-Level Parallelism (STLP)

**Source:** Akkary & Driscoll, "A Dynamic Data Dependence Speculative Multithreading Architecture" (ISCA '98); Steffan & Mowry, "The Potential for Using Thread-Level Data Speculation" (1998); Oracle Labs, "GraalVM Truffle FastPath" (2023)

**The Idea:** For hot sequential loops where the JIT can't prove independence at compile time, **speculatively execute loop iterations in parallel on other cores**. If a cross-iteration dependency is detected at runtime, discard the speculative result and re-execute sequentially.

**Why It's a Bomb:** This is the holy grail for single-threaded-looking code that has hidden parallelism. No mainstream JIT does it. It effectively gives you multi-core speedup on code that was written sequentially, with zero programmer effort.

**Implementation:**
```
1. Profile identifies hot loop with high iteration count (>1000 trips)
2. Run dependence analysis:
   - If provably independent: direct parallelization (no speculation)
   - If ambiguous: speculatively parallelize with runtime checks
3. At loop entry:
   a. Fork N-1 worker threads (where N = core count)
   b. Each worker processes iterations [start..end] speculatively
   c. Write buffer: all stores go to a per-thread write buffer (not main memory)
   d. Validation: after all workers finish, check that no two workers wrote to the same address
   e. If valid: commit write buffers to main memory
   f. If conflict: discard speculative results, re-run sequentially
4. Speculation success rate tracked per-loop; if <50%, stop speculating
```

**Expected gain:** Up to N-core speedup on parallelizable loops. Even with 70% speculation success rate on 4 cores, expected 2.5x. Zero overhead on non-parallelizable code (just the initial check).

**Scalability:** Thread pool is pre-created. Only hot loops with >10K backedges get this treatment. Write buffer size is bounded (one cache line per store, up to 4KB per worker).

### BOMB 4: BOLT-Style Post-JIT Binary Optimization

**Source:** Maksim Panchenko et al., "BOLT: A Practical Binary Optimizer for Data Centers" (Facebook, 2019)

**The Idea:** After JIT code is generated and profiled, do a **second full compilation pass** that treats the already-generated native code as input and reoptimizes it with exact profile data. This includes:
- **Code layout reordering** (Pettis-Hansen) for optimal icache behavior
- **Cross-function register allocation** (callee-saved become caller-saved for hot pairs)
- **Dead code elimination** across function boundaries
- **Branch alignment** for the specific microarchitecture's branch predictor

**Why It's a Bomb:** Facebook's BOLT showed **5-15% speedup on already -O3 compiled C++ code**. On JIT code, which is typically less optimized than -O3, the potential is even higher. The key insight: JIT code suffers from **fragmented code layout** because methods are compiled independently. BOLT-style reordering fixes this.

**Implementation:**
```
1. After T2/T3 has compiled a working set of methods and collected PMU profiles:
2. Build a hot-call-graph from profile data
3. Apply Pettis-Hansen chain-based layout:
   - Start with the most frequent edge (caller→callee)
   - Chain functions together so hot paths are contiguous in memory
   - Cold paths are moved to separate "cold" code segments
4. Re-allocate registers across hot function pairs:
   - If f() always calls g(), turn callee-saved into caller-saved between them
   - Eliminate push/pop sequences for registers that g() doesn't use
5. Align hot loop headers to 32-byte boundaries (reduces decode stalls)
6. Re-emit the entire code cache as a single optimized binary
```

**Expected gain:** 5-15% on large codebases with many virtual calls. Up to 30% icache miss reduction.

### BOMB 5: Hardware Performance Counter (PMU) Feedback Loop

**Source:** Chen et al., "Profile-Driven Optimizations for JavaScript" (CGO '18); Ithemal (MIT, 2019)

**The Idea:** Use **actual CPU performance counters** — L1d miss rate, L1i miss rate, branch mispredictions, TLB misses, uop stalls — to drive recompilation decisions. No other JIT does this. They all use software counters (branch taken/not-taken, type frequencies). Hardware counters tell you **why** code is slow, not just **which** code is slow.

**Implementation:**
```
1. After T2 compilation, attach a PMU monitoring thread that periodically samples:
   - PERF_COUNT_HW_CACHE_MISSES (L1d)
   - PERF_COUNT_HW_BRANCH_MISSES
   - PERF_COUNT_HW_INSTRUCTIONS vs CYCLES (IPC)
   - PERF_COUNT_HW_STALLED_CYCLES_BACKEND
2. Every 100ms, check if any compiled method has:
   - IPC < 1.0 (likely memory-bound or stall-bound)
   - High L1d miss rate → insert software prefetches, apply data layout optimization
   - High branch mispredict → convert branches to cmov, widen guards
   - High backend stalls → recompile with different scheduling (avoid port conflicts)
3. Recompile with PMU-informed decisions as T3.5
```

**Expected gain:** 5-20% on memory-bound and branch-heavy workloads. This is the optimization that no static compiler can do — it requires runtime hardware feedback.

### BOMB 6: Context-Sensitive Multi-Versioning with Partial Evaluation

**Source:** The Futamura Projections (1971); GraalVM Truffle (2014); warmup-less JIT research (2019+)

**The Idea:** Don't compile one version of a method. Compile **multiple versions**, each specialized to the actual calling context. This is partial evaluation applied at the method level: if `sort(array)` is always called with an `Int32Array` from `main()` and a `StringArray` from `parse()`, compile two versions — one with no type checks and direct int32 element access, another with no type checks and direct string operations.

**Why It's a Bomb:** This is what makes GraalVM's Truffle fast enough to **match or beat hand-written C interpreters** for guest languages. The specialization eliminates every dynamic dispatch, every type check, every branch that's always one way in a given context.

**Implementation:**
```
1. Track call-site → callee → receiver-type triples in type feedback
2. For hot call sites (>10K calls):
   a. Identify the top-K receiver types at this call site (K ≤ 4)
   b. For each top type, create a specialized version of the callee:
      - Replace virtual dispatch with direct call (or inline)
      - Replace type checks with Unconditional guards
      - Replace field access with fixed-offset loads (no shape lookup)
      - Propagate constant arguments (e.g., if arg2 is always 42)
   c. At the call site, emit a type-check switch:
      if (receiver_type == Int32Array) call sort_Int32Array(...)
      else if (receiver_type == StringArray) call sort_StringArray(...)
      else call sort_generic(...)
3. Track per-version execution count; cold versions get evicted from code cache
```

**Expected gain:** 15-40% on polymorphic OO code. This is the single biggest win for "real-world" programs with complex type hierarchies.

### BOMB 7: Allocation Elimination via Store-to-Load Forwarding

**Source:** Click et al., "Escape Analysis for Java" (OOPSLA '99); Kotzmann et al., "Design of the Java HotSpot Client Compiler" (2008)

**The Idea:** Extend PEA beyond "does this object escape?" to "can we eliminate **every memory operation** involving this object?" Current PEA says: if the object doesn't escape, replace allocation with scalars. But even for **escaping** objects, if you can observe that a store to `obj.field` is always followed by a load from `obj.field` with no intervening alias, replace the load with the stored value.

**Combined with allocation:** Allocate → Store(x) → Load → use. Replace with just `use x`. The allocation, store, and load all disappear.

**Implementation:**
```
1. After PEA, run a Store-Load Forwarding (SLF) pass on the SoN IR:
   a. For each LoadField node, find the nearest dominating StoreField on the same object+field
   b. Check: no intervening call that may alias, no intervening StoreField on same field
   c. If safe: replace LoadField with the StoreField's input value
   d. If the StoreField's value is a constant → constant propagation
   e. If the object is scalar-replaced → field access becomes register access
2. Run DCE after SLF to eliminate dead StoreField nodes (stores whose loads are all forwarded)
3. Run DCE again to eliminate dead allocations (allocations whose fields are all dead stores)
```

**Expected gain:** 10-30% on allocation-heavy code. Combined with PEA, this can eliminate **80-90% of all allocations** in tight numeric loops (the pattern `new Point(x,y); distance += p.x * p.x + p.y * p.y` becomes just the arithmetic).

### BOMB 8: Object Inlining (Object Fusion)

**Source:** Wimmer et al., "Object Inlining for Java" (VEE '10); Ibrahim et al., "Object Inlining in the Jikes RVM" (2013)

**The Idea:** When object A contains a field pointing to object B, and B is **only ever accessed through A**, fuse them into a single allocation. B's fields become additional fields of A, laid out contiguously. This eliminates one allocation, one pointer dereference, and improves cache locality.

**Example:**
```java
class Point { double x, y; }
class Circle { Point center; double radius; }

// Before: Circle = [header, center_ptr→, radius]
//         Point  = [header, x, y]
// After:  Circle = [header, x, y, radius]  (fused, no Point allocation)
```

**Implementation:**
```
1. Build an allocation graph (already exists in PEA's cross-object SR)
2. For each edge A.field → B where B is only allocated at this site:
   a. Check that B is only accessed through A (no alias escapes B independently)
   b. Compute the fused layout: A's fields + B's fields (skip B's header)
   c. Rewrite A's allocation to the fused size
   d. Replace A.field access with direct access to the fused offset
   e. Replace B.field access with A access at the fused offset + B.field_offset
3. Deopt: at materialization points, split the fused object back into A + B
```

**Expected gain:** 10-25% on object-heavy code (compilers, interpreters, AST walkers). This is especially impactful for tree-structured data.

### BOMB 9: Value Numbering Across Guard Boundaries (Guard-Aware GVN)

**Source:** Click, "Global Code Motion / Global Value Numbering" (PLDI '95); improved by Pop et al., "SSA-based GVN with Predicates" (2011)

**The Idea:** Current GVN treats `Guard(x is Int)` and `x` as different values. But after a guard succeeds, **you know x is an Int**, which means:
- `vtx_is_smi(x)` is always true → eliminate the check
- `vtx_smi_value(x)` is safe → no tag extraction needed, use raw value
- Any subsequent `Guard(x is Int)` is redundant → eliminate

This is **constraint propagation through guards**: after `Guard(type == T)`, all downstream nodes inherit the constraint `type == T`. This is what V8's "typed lowering" does, but VORTEX can do it better because SoN IR makes guard dependencies explicit.

**Implementation:**
```
1. Extend GVN to carry type constraints:
   - After Guard(x, type=Int), record in a type-map: {x → Int}
   - When encountering vtx_is_smi(x) downstream, check type-map: if x→Int, replace with true
   - When encountering vtx_smi_value(x) downstream, replace with an UnboxInt(x) node that's free
2. Extend to range constraints:
   - After Guard(i < len), record: {i → [0, len)}
   - When encountering Guard(i < len) again → Unconditional (eliminate)
   - When encountering ARRAY_LOAD(a, i) → eliminate bounds check
3. This subsumes bounds check elimination and null check elimination as special cases
```

**Expected gain:** 5-15% on heavily-guarded code. This eliminates the "guard tax" that makes JIT code slower than C.

### BOMB 10: Probabilistic Call Graph + Bayesian Inlining

**Source:** Chen et al., "Bayesian Optimization for Compiler Phase Ordering" (2021); Cummins et al., "CompilerGym" (2021); V8's ML-based inlining (2022)

**The Idea:** Replace the current GBDT inliner with a **full probabilistic model** of the call graph. At each call site, don't just ask "should I inline?" — ask "given everything I know about this program's behavior, what inlining decision maximizes expected future performance?" This requires modeling:
- The probability that inlining enables further optimizations (PEA, constant prop, guard elimination)
- The probability that inlining causes register pressure that spills
- The probability that the call target changes type profile in the future
- The expected deopt rate

**Implementation:**
```
1. Build a probabilistic call graph:
   - Each edge has: call frequency, receiver type distribution, arg constness
   - Each node has: size, complexity, has_loops, has_allocations, has_calls
2. At inlining decision time:
   - Compute P(enables_optimization | inline) using a learned model
   - Compute P(causes_spill | inline) using register pressure model
   - Compute E[performance_delta] = P(opt) * benefit - P(spill) * cost - size_cost
   - Inline if E[delta] > 0
3. After inlining, measure actual outcome and update the model (online learning)
4. Use Thompson sampling for exploration-exploitation: sometimes inline uncertain cases to learn
```

**Expected gain:** 3-10% over GBDT inliner. The real win is **stability**: probabilistic models don't have the sharp threshold effects that decision trees do, so you avoid the "inline cascade" problem where one bad inline decision causes a cascade of spills.

### BOMB 11: Data Layout Optimization (Runtime-Driven Field Reordering)

**Source:** Hogg et al., "Jikes RVM Field Reordering" (2013); Chabbi et al., "An Efficient Dynamic Optimizer for HPC" (2022)

**The Idea:** Track which fields are accessed together (temporal co-access patterns) at runtime. Then reorder fields in memory so that co-accessed fields share cache lines. This is impossible for a C compiler (it doesn't know the access patterns at compile time), but trivial for a JIT.

**Implementation:**
```
1. During T0/T1, record co-access vectors per type:
   - Every LoadField(obj, f) also records: "within 100 instructions, also accessed fields f2, f3"
   - Build a co-access matrix: C[f1][f2] = number of times f1 and f2 were accessed close together
2. At T2 compilation, for each type with hot allocation sites:
   a. Solve a traveling-salesman-like problem: order fields to minimize total co-access distance
   b. Create a new "layout version" of the type with the reordered fields
   c. Allocate new objects with the new layout; existing objects use the old layout
   d. Generate shape-id-based dispatch in JIT code: if shape_id == NEW_LAYOUT, use new offsets
3. Old-layout objects are migrated lazily during GC (copy to new layout)
```

**Expected gain:** 10-30% on pointer-chasing workloads (compilers, databases, graph algorithms). Cache miss reduction of 20-40%.

### BOMB 12: Snapshot-Based Deoptimization (LuaJIT-Style)

**Source:** Mike Pall, "LuaJIT 2.0 Internals" (2012); "The Interpreter Problem" blog series

**The Idea:** Instead of building full `FrameState` nodes at every potential deopt point (which is expensive in compilation time and memory), use **compressed snapshots**: record only the *differences* from the previous snapshot. At deopt time, walk the snapshot chain to reconstruct the full state.

This is how LuaJIT achieves its famously fast compilation speed while still having deopt. Current VORTEX creates a `FrameState` node per deopt point, each containing full locals+stack+monitors. That's O(N*F) where N=deopt points, F=frame size. Snapshots are O(N*delta) where delta is typically 2-3 entries.

**Implementation:**
```
1. Instead of FrameState nodes, emit snapshot records:
   - A snapshot = {bytecode_pc, diff_entries[]}
   - Each diff_entry = {slot_index, value_source} where value_source is:
     * Const(value) — literal constant
     * Reg(vreg) — from physical register
     * Stack(offset) — from stack slot
     * SameAs(prev_snapshot_slot) — unchanged from previous snapshot
2. At deopt:
   a. Start from the base snapshot (method entry)
   b. Apply diffs in order to reconstruct the interpreter frame
   c. Jump to the interpreter at the recorded bytecode_pc
3. Snapshots are emitted as a compact side table (no IR nodes needed)
```

**Expected gain:** 20-40% faster compilation time, 50-70% less deopt metadata memory, and enables more aggressive inlining (deopt cost is lower).

### BOMB 13: Adaptive Stack Frame Layout (Profile-Guided Frame Packing)

**Source:** Novel — I haven't seen this in any published JIT, but the idea is straightforward.

**The Idea:** Not all locals are equal. Some are accessed 1000x per invocation (loop counters), some are accessed once (setup variables). Currently, locals are laid out sequentially: local0 at [rbp-8], local1 at [rbp-16], etc. This means hot locals may need 4-byte displacements from RBP while cold locals that are never accessed also get stack space.

**Optimize:**
1. Sort locals by access frequency (from profile data)
2. Place the 4 most-accessed locals at [rbp-8] through [rbp-32] (1-byte displacement in x86 ModR/M)
3. Place next 4 at [rbp-40] through [rbp-64]
4. Eliminate locals that are never read (already dead — DCE should catch this, but some survive due to conservative analysis)
5. Share stack slots between non-overlapping live ranges (like register allocation for stack)

**Expected gain:** 2-5% on methods with many locals (common in compiled-to-bytecode languages). The real win is code size: shorter displacements = smaller instructions = better icache utilization.

### BOMB 14: Rematerialization-Aware Register Allocation

**Source:** Briggs et al., "Rematerialization" (PLDI '92); Wimmer & Franz, "Linear Scan Register Allocation on SSA Form" (CGO '10)

**The Idea:** When the register allocator needs to spill a value, instead of always spilling to memory (store + load = 2 memory accesses), **rematerialize** the value if it can be recomputed more cheaply than loaded from memory.

**Examples of cheap rematerialization:**
- Constants: `mov rax, 42` (7 bytes) vs. `mov rax, [rsp+24]` (5 bytes but a cache miss)
- Loop counters: `lea rax, [rcx+1]` (3 bytes) from the compare operand
- Base addresses: `lea rax, [rbp-32]` (4 bytes) from the frame pointer
- Tagged constants: `mov rax, 0x7ff8000000000000` (NaN-box header) vs. spilling

**Implementation:**
```
1. During linear scan, when a spill is needed:
   a. Classify each value's rematerialization cost:
      - TRIVIAL: constant, frame address (1-2 instructions)
      - CHEAP: derived from live values (1-3 instructions)
      - EXPENSIVE: loaded from memory, call result (spill is cheaper)
   b. If rematerialization cost < spill cost (2 memory ops), rematerialize instead
   c. Emit the rematerialization at the point of use instead of a load
2. Split live ranges at rematerialization points to avoid keeping the value live across its entire range
```

**Expected gain:** 3-8% on register-pressure-heavy code (after aggressive inlining). This directly addresses the biggest cost of inlining: register pressure causing spills.

### BOMB 15: Continuous Background Recompilation with Zero-Pause Installation

**Source:** Already sketched in `sota/recomp.h`, but let's push it further. Based on Azul's C4/Zing approach.

**The Idea:** The JIT should **never** block the mutator for compilation. Ever. Currently, when T2 triggers, the mutator might stall waiting for compilation. Instead:

1. **Background compiler threads** continuously recompile based on latest profile data
2. **Code installation is lock-free**: new code is published via an atomic pointer swap at the method's code entry point
3. **Transition is at the next safepoint**: currently-executing old-code frames finish their invocation; new invocations use the new code
4. **Multiple versions coexist**: old code stays alive until all frames using it return

**The key Azul innovation**: Use a **read barrier** on method dispatch. The inline cache checks a version counter. If the counter changed, the IC self-patches to the new code. This is a single comparison on every call — nearly free.

**Expected gain:** Eliminates JIT pause time entirely. For latency-sensitive applications, this alone can be worth 50-100% improvement in tail latency (P99).

## What Makes VORTEX Uniquely Positioned

The SoN IR architecture is the **key differentiator**. Most of these optimizations are hard or impossible on basic-block or CFG-based IRs because they require reasoning about values across control flow. SoN makes values first-class — a `Constant(42)` node has no control dependency and can be placed anywhere. This means:

- **GVN is trivial** (already implemented)
- **PEA is natural** (already implemented)
- **Store-Load Forwarding** is a simple dominator check
- **Guard-Aware GVN** is a type-map propagation
- **Object Inlining** is a graph rewrite
- **Polyhedral opts** apply to the loop subgraph

The other key advantage: **trace-based region formation**. By compiling only the hot path (with guards for cold paths), you naturally get small compilation units that are amenable to aggressive optimization without exploding compile time.

## The One Optimization That Beats Them All

If I had to pick **one** bomb to implement first, it's **Speculative Untagging with Left-Shifted SMI** (from my previous message) combined with **Guard-Aware GVN** (Bomb 9). Together, they make tagged arithmetic essentially free:

```
Before: IADD → check tags → extract values → add → re-encode → check overflow
After:  IADD → add (that's it — the add IS the tag operation with left-shifted SMI)
        + single overflow guard (hoisted out of loops by guard-aware GVN)
```

This pair alone closes 60-70% of the gap to C on numeric code. Everything else builds on top of it.
